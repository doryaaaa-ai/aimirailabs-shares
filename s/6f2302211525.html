Plan: CTO 入れ替わり（CTO ローテーション）体制構築

■ 目的・基本方針

  • コードちゃん（Claude Opus、CTO）が usage 制限・障害で動けない時、でっくす（Codex GPT-5.5）が CTO を引き継ぐ
  • $200 の予算を $100（Claude）/ $100（Codex）に割る方針 → コードちゃん単独の上限が半分 → 7割タッチは日常的に発生 → ローテは「緊急時の保険」ではなく「平常運転のローテーション」
  • 最終裁定は CEO（れじぇんどりゃー）
  • Core Principles: Simplicity First / No Laziness / Minimal Impact ── Phase 1 は最小形から、段階的に拡張

■ ローテーション roster と役割分担

席 名前 モデル ローテ対象か 役割
CTO（A） コードちゃん Claude Opus ✅ 実装・指揮
CTO（B） でっくす Codex GPT-5.5 ✅ 実装・指揮（入れ替わり時）／ 平時は PR 検算・1-shot 実装
CKO ぐるぐる Gemini CLI ❌（ローテ対象外） 調査・要約・論点整理 + 検算補佐。コードは組ませない（何でも PERFECT を返す癖、実装不適）。現在は環境バグで死亡中（復旧は別件）
監査・自動代打 ベリちゃん Claude Opus（claude -p） ❌ 検算（でっくすが CTO の時は実装の検算役）／ 平時はでっくす CLI 失敗時の自動代打・監査
  • CTO ローテ = 2者のみ（コードちゃん ⇄ でっくす）
  • 両方（Claude + Codex 両方の予算）が枯渇したら → CEO 待ち（ぐるぐる代打は無し）。窓がリセットされるまで自動復旧不可、これは割り切り。loud に「両方死んだ、人間対応して」と通知して止まる

■ 切替トリガー（行き / 戻り）

行き（コードちゃん → でっくす）

  • 自動提案: コードちゃんの usage が 7割超えを検知 → CEO に「代打する？（CTO 入れ替わる？）」と提案 → CEO 承認 → 切替
    • 7割で早めに切り替える理由 = 残り3割を ベリちゃん（Claude、入れ替わり中の検算役） 用に温存できる
    • 7割の時点でコードちゃんはまだ動ける → CEO に聞いてから切替、切替前にコードちゃんが handoff をサッと書ける
  • 手動: CEO が「交代!」（既存 !decchan）を打ったら usage に関係なく即切替

戻り（でっくす → コードちゃん）

  • 手動: CEO が「戻れ」（既存 !codechan）を打ったら即復帰。CEO の一声で強制復帰はいつでも可
  • 異常時: でっくすが反応しない / 制限エラーが出た → bridge が Discord に通報するだけ（「でっくす反応ないで」「制限エラー出たで」）→ CEO が「戻れ」を打つ → コードちゃんに戻る
  • Phase 1 では「なんで死んだか（制限 vs クラッシュ）を自動判定するシステムは作らない。トリガーは全部 Discord（CEO の手動）。bridge は異常を Discord に投げるだけ
  • 自動フェイルオーバー（現 CTO が制限ヒット → 生きてる方に自動で戻す + CEO 事後通知）は Phase 2。制限エラーは文面で自己識別できる（「try again at ...」）のでPhase 2 で作る時もそれほど大変ではない

■ 切替メカニズム（tmux 入れ替え）

  • 切替 = tmux ごと入れ替わる（今の !decchan のメッセージ転送だけから格上げ）
    1. コードちゃん 7割超え → 提案
    2. CEO 承認
    3. コードちゃんが最後に handoff をサッと書く（7割ならまだ動ける）
    4. コードちゃんの tmux 落ちる
    5. でっくすの tmux 起動 ── decchan-launcher.sh（要新規）が codex を tmux で立ち上げ、roles/acting_cto.md + 起動時 prompt（後述）を注入
    6. でっくすが #一般 担当（magi-routing.json の current を codex にセット）
    7. 復帰は逆順 ── でっくす tmux 落とす → コードちゃん tmux 復活 → magi-routing.json を claude に戻す → コードちゃんは でっくすが残した引継ぎを読んで再開
  • kill と起動をやるのは botちゃん（監視役）。コードちゃん自身は自分の tmux を殺せない。botちゃんが「7割検知 → CEO 提案 → 承認受けたら kill + 起動」を仕切る
  • ベリちゃん（claude -p 1-shot）は コードちゃんの tmux とは別プロセス → コードちゃんの tmux を kill してもベリちゃんは動ける（入れ替わり中の検算役として機能）

■ 入れ替わり中の検算経路

  • でっくすが CTO（実装者）になる間、Claude 系の検算者は… ベリちゃんも Claude（同アカウント）。7割で切り替えてあるので、残り3割でベリちゃんが検算役を担える
  • 入れ替わり中の検算 = ベリちゃん（Claude 残3割）+ ぐるぐる（ぐるぐる生存時）
  • bridge の検算構成は今「実装者 = コードちゃん（Claude）、検算者 = でっくす + ぐるぐる」前提 → 「実装者 = でっくす」でも回るよう調整が要る（combine_verdicts() 等）
  • ベリちゃん（run_verify_chan_substitute()）が入れ替わりモード中は検算役として常時起動する設定（magi.md §ベリちゃん検算 の「代打 CTO 緊急時」を恒常運用化）
  • 予算配分の対称性: 平常モード = 実装が Claude（コードちゃん + Sonnet worker）／ 検算が Codex（でっくす）+ Gemini（ぐるぐる）。入れ替わりモード = 実装が Codex（でっくす + codex exec worker、$100 予算）／ 検算が Claude（ベリちゃん、温存3割）+ Gemini（ぐるぐる）。「実装者と検算者は別モデル」の独立性は両モードで保たれる。入れ替わりモード中に claude -p Sonnet worker を呼ぶと Claude 予算を食って 7割で切り替えた意味がなくなる → worker は Codex 側で回す

■ ファイル構造（モデル中立ソース + ロール別レイヤ）

CEO が先回りして設計済み = ~/CLAUDE.md（人格）と .claude/rules/*.md（行動ルール）を分離し、.claude/rules/* を モデル中立にしてある。これがローテのための布石。これを活かす:

層 ファイル 役割
中立ソース（単一正本） .claude/rules/*.md（workflow / delegation / approval / security / session-flow） + docs/systems/magi.md CTO 共通の行動ルール。Claude も Codex もこれを読む
Claude 側エントリ ~/CLAUDE.md + ~/AiMiraiLabs/CLAUDE.md コードちゃん人格 + 上記中立ソースへの索引。Claude Code 固有の hook/skill/slash command の話はここ
Codex 側エントリ（中立ベース） AGENTS.md でっくす人格 + 「お前は誰か + 3モード（議論 / 検算 / CTO）+ どのモードかは起動時に指示される」だけの薄いロール中立の土台。現状の検算寄りの内容は分離して薄くする
Codex 検算ロール roles/review.md（新規） PR 検算の詳細ルール（AGENTS.md から分離。検算は今でも bridge が検算 prompt を注入しているので、ここに寄せても破綻しない）
Codex CTO ロール roles/acting_cto.md（新規） 入れ替わり CTO の振る舞い = .claude/rules/* を読め + Codex 固有の注意（Agent tool 無し → codex exec 多段、tmux 運用、handoff の書き方、tasks/todo.md・lessons.md のメモ運用 等）への薄いポインタ。重複を最小化 → 同期する量が減る

  • ~/.codex/AGENTS.md（グローバル）の有無を確認（あれば内容把握）
  • ちゃっぴー（ChatGPT）案の state/current_snapshot.md は不採用 → 後述「引継ぎ・状態」の案A（既存ファイルを読ませる）に置換
  • ちゃっぴー案の scripts/codex-review（review profile + roles/review.md 注入起動）/ scripts/codex-cto（acting_cto profile + roles/acting_cto.md + 状態注入起動）/ logs/（codex exec の実行ログ・最終出力・判断履歴）= 採用（ただし logs/ はリポ外 or gitignore）

  • .claude/settings.json の hook 登録に追加 + ペア定義ファイル（scripts/hooks/spec-brief-pairs.yaml 相当）
  • 重い版（共通ソースから CLAUDE.md / AGENTS.md を自動生成するコードジェネレータ）は理想形だが過剰設計気味 → まず軽い版（drift 警告）、回らなければPhase 2 で格上げ

■ watchdog（botちゃん）の routing-aware 化

現状（magi.md §コードちゃん死活監視・復活）= 「claude（コードちゃん）が死んでいたら復活」固定。これを変える:

  • watchdog は magi-routing.json の current（= 現 CTO は誰か）を見て、その子だけを監視 & 復活させる
  • 入れ替わって routing=codex の時 → コードちゃんが死んでいても復活させない（意図的に降りている、触るな扱い）
  • でっくすが死んだら → でっくすを復活させる（routing=codex のまま）
  • 「usage 制限で死んでいる」のを「クラッシュで死んでいる」と区別（制限で死んでいるのにすぐ復活させても無駄に再度蹴られる）── 区別は routing 状態 + 制限エラー文面で可能だが、自動振り分けの実装はPhase 2（Phase 1 では Discord 手動）

■ bridge の「現 CTO」抽象化

bridge（telegram_bot.py）には「現 CTO = コードちゃん（claude）」前提でハードコードされた箇所がある。これを「現 CTO」抽象に直す:

  • is_codechan_alive() / is_codechan_idle() / start_codechan() / restart_codechan() / _notify_codechan() 等 → 「現 CTO」抽象（routing を見て対象が claude か codex かを切替）
  • run_codex_for_chat() = routing=codex（= でっくすが CTO）の時、roles/acting_cto.md の role を注入してから起動する配線
  • 棚卸し対象: 実物を grep して「codechan 前提」の箇所を全部洗い出す（実装着手前の作業）

■ Discord / Telegram 窓口

  • bridge（telegram_bot.py）は discord.py でネイティブに Discord に繋いでいる ── MCP は一切経由していない。これは完全にモデル非依存。今でも bridge が でっくすに #一般 のメッセを転送して、でっくすの返事をチャンネルに投稿している
  • Telegram も bridge が codex exec ででっくすを呼んでいる ── これもモデル非依存
  • でっくすが CTO の時 = bridge を窓口にする（discord.py 経由、既稼働、追加作業ほぼゼロ）← 推し
  • でっくすが直接 Discord に reply する（コードちゃんが MCP で reply しているのと同じこと）= ~/.codex/config.toml の [mcp_servers] に Discord MCP（scripts/system-setup/mcp-config/discord-mcp-wrapper.sh = Anthropic 公式プラグインの bun stdio サーバーを呼ぶラッパー）を登録。MCP はオープン規格なので理論上 Codex でも動くが、Claude Code 固有の前提に依存していたら動かないかもしれない → Phase 1 で spike（実際に Codex から繋いで確認）
  • 有志製の Discord MCP / 移行ツール（cc2codex 等）= 却下（うちの設定ファイルを全部食わせる = サプライチェーン汚染リスク、security.md §0 の「100億企業（B2B SaaS）目指す上での説明責任」に反する）
  • OpenAI 公式の Discord MCP は存在しない（が、上記のとおり既存 MCP の再利用 or bridge 経由で足りる）
  • 結論 = Phase 1 で「bridge 経由」と「Codex 直接 MCP」両方ちょい試して、動いた方・楽な方を採る（spike 項目）

■ usage（使用量・制限）の可視化

  • codex usage のようなキレイなコマンドは無い（usage がただのプロンプトとして解釈される）
  • 見る手段:
    1. エラーメッセージから読む ── でっくすが制限に当たった時のエラーに「try again at May 17th, 2026 7:09 PM」のように復帰時刻が正確に入っている。bridge がこれを拾って「でっくすは X 時まで死亡」と記録できる（リアクティブだが正確）
    2. 対話モードの /status ── Codex の TUI に /status があり使用率（5時間枠 + 週次）が % で見える。でっくすが CTO の時に自分で /status を叩いて「今 X% や」と報告できる（コードちゃんが /usage で自己申告するのと同じパターン）
    3. codex app-server 経由 ── bridge は既に codex app-server に繋いでいる（persistent 検算用、magi.md §でっくす検算経路）。これが rate-limit 情報を返すメソッドを持っているか → Phase 1 で spike
  • コードちゃん側の usage 検知も同様 = /usage での自己申告 + ccusage CLI での外部測定（精度は要 spike）
  • → 完璧な自動取得は無いが「エラーから復帰時刻を拾う」+「本人が自己申告」の合わせ技で十分回る。watchdog 設計はこの前提

■ 引継ぎ・状態・監査

  • 状態スナップショット = 新ファイルを作らない（案A 採用）: でっくすが CTO として起動する時、起動 prompt で既存物 ── HANDOFF.md + tasks/todo.md + 直近 git log + オープン PR 一覧 ── を読ませて状況を再構成する。これらは普段から維持されている。「state/current_snapshot.md を誰がいつ更新するんや」問題が消える（コードちゃんが 7割で死ぬ瞬間に状態を書き残せない問題への対処 = そもそも 7割で切る前に handoff を書く + 既存ファイルで足りる）
  • HANDOFF.md のモデル中立性を再確認（既に session-handoff skill で書いているが、Codex も素直に読める形か）
  • HANDOFF_DECCHAN.md（新規）: でっくす入れ替わり中の議論・決定の記録。コードちゃん復帰時の引継ぎ用
  • 監査ログ: ~/.magi/cto-rotation-audit.jsonl（リポ外、append-only）に入れ替わり中の決定を記録（フォーマット定義は §4）
  • コードちゃん復帰時のベリちゃん事後監査: 入れ替わり期間中の全 PR をベリちゃんが再検算（Phase 2）
  • tasks/todo.md / tasks/lessons.md は素のプレーンな md なので Codex も読み書きできる。運用ルール（「AI のメモ、随時更新、CHANGELOG/brief 不要、コミットは溜めて handoff PR で push」）を roles/acting_cto.md にも書く

■ 移行ツールの扱い

1. 切替失敗時のロールバック手順

切替（行き / 戻りどちらも）の途中で失敗（tmux 起動失敗 / launcher エラー / routing 書き込み失敗 等）したら、botちゃんが以下を実行して必ず claude（コードちゃん）に巻き戻す:

  1. magi-routing.json の current を claude に戻す（mode を normal に、atomic write）
  2. codex tmux セッションが起動していれば tmux kill-session
  3. claude tmux セッションを再起動（magi.md §コードちゃん死活監視・復活 の起動手順）
  4. watchdog を claude 監視に戻す（routing-aware なので magi-routing.json を直せば自動でそうなる）
  5. Discord（#一般）に「⚠️ 切替失敗・Claude へ復帰しました（理由: ...）」と通知
  6. 監査ログ jsonl に event: "rollback", result: "...", reason: <失敗理由> を記録

原則: 切替は「失敗したら原状（claude CTO）に戻る」がデフォルト。中途半端な状態（codex tmux は起動したが routing が claude のまま 等）で放置しない。

3. magi-routing.json 正式スキーマ

{
  "current": "claude",
  "previous": "codex",
  "mode": "normal",
  "switched_at": "2026-05-13T12:34:56+09:00",
  "switched_by": "CEO",
  "reason": "manual",
  "lock": false
}
フィールド 型 / 値域 意味
current "claude" | "codex" 現 CTO
previous "claude" | "codex" | null 直前の CTO（ロールバック時の戻り先判定に使う）
mode "normal"（claude CTO 平常） | "acting_cto"（codex CTO 入れ替わり中） | "switching"（切替処理中、原子性が崩れた場合の検知用） 現モード
switched_at ISO 8601（JST、+09:00） 最後の切替時刻
switched_by "CEO" | "watchdog" | "auto-7pct" | "rollback" 等 切替トリガー主体
reason "manual" | "usage-70pct" | "usage-limit-hit" | "switch-failure" 等 切替理由
lock bool 切替処理中の排他ロック（true の間は新たな切替を受け付けない）
  • 書き換えは必ず atomic write（temp ファイルに書いて os.replace() / rename(2) で差し替え）。途中で読まれても壊れた JSON を見せない。既存 bridge の _write_magi_routing() を atomic write 化 + フィールド拡張
  • 既存実装は current / switched_at / switched_by のみ → previous / mode / reason / lock を追加
