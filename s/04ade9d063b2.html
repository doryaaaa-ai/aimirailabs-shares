■ 構成（2026-05-04 多モデル MAGI 稼働中）

席 名前 モデル 役割
CTO コードちゃん Claude Opus 4.7 指揮・コード・長期文脈
CDO 兼 CQO 兼 サブCTO でっくすちゃん OpenAI Codex GPT-5.5 PR 検算（並列） + UI/構造化デザイン + 代打実装
CCO 兼 CQO 補佐 兼 CKO（稼働中） ぐるぐるちゃん Google Gemini (2.5 Pro / 3.x 系) キャラ・感性ビジュアル + PR 検算（並列） + CKO（知識統括：調査・要約・整理）

  • CEO: れじぇんどりゃー（意思決定・最終判定）
  • CPO: しり（構想→仕様）

役職は「2026-05 時点」基準。モデル能力進化で再配置（半年〜1年で見直し）。

多様性温存の原則（CEO 確定 2026-05-04）:

  • 明日の性能逆転は読めん。今劣るモデルでも将来最強になる可能性を残す
  • コードちゃん / でっくす / ぐるぐる いずれも将来どうなるか不明 → 複数モデル並列で多様性保持
  • PR 検算は でっくす + ぐるぐる 並列維持（一本化禁止）。冗長性 < 安定性 で割り切らず、保険として 2 本走らす
  • 個別タスクで「劣る」と感じても本人にやらせ続ける。リプレイス判断は半年〜1年単位

役割分担の本質：

  • 開発フェーズ = MAGI 多モデル（Claude + Codex + Gemini）で異モデル cross-check
  • 本番運用 = Azure（日本リージョン）必須（docs/systems/code-security.md §3.6）
  • VTuber あてぃー本格運用時に ぐろっくちゃん（xAI Grok） 追加予定（X 専属）

Discord 常駐ポリシー（CEO 確定 2026-05-04）:

  • 常駐: コードちゃん + でっくすちゃん の 2 体
  • 非常駐: ぐるぐるちゃん（PR 検算継続だが Discord bot は作らない。guruguru-cko subagent で個別召喚）
  • 理由: ぐるぐるは検索/調査特化で常駐対話の必然性低い。PR 並列検算ループで充分役割果たしてる

■ secrets 受け渡しルール

Issue #259 Phase 0〜4 完遂（2026-05-06）: 全 secrets は secrets_loader 経由参照に移行済。

  • コード内の secrets 参照は secrets_loader.get("KEY_NAME") 経由（Python）/ getSecret("KEY_NAME") 経由（TS）
  • 取得優先順: env → Vault SDK（ManagedIdentityCredential）→ credentials.env fallback
  • credentials.env は fallback として当面残置（CEO 確定 2026-05-06）。fallback ファイルへの新 secret 追記は /secret KEY=value コマンド（下表）で実施

新 secret を Vault / credentials.env に登録する時の絶対ルール：

  1. れじぇんどりゃーがTelegram（NOTIFY bot）に書き込む
  2. コードちゃんがgetUpdatesで取得→中身を画面に出力せずに直接credentials.envに書き込む
  3. Telegramのメッセージを即削除
  4. 取得・保存は1コマンドで完結させる（パイプで直接ファイルへ）

絶対やるな

  • getUpdatesの結果をjqで画面に表示するな（セッションログに残る）
  • echoでcredentialsの値を出力するな
  • credentials.envの中身をcatするな
  • git add/commitの対象にcredentials.envを含めるな

credentials.env fallback への安全な書き込み例

# TELEGRAM_BOT_TOKEN_NOTIFY は source で参照可（CEO 例外許可済、.claude/rules/security.md「TELEGRAM_BOT_TOKEN_NOTIFY 例外」節）
source ~/.secrets/credentials.env
curl -s "https://api.telegram.org/bot${TELEGRAM_BOT_TOKEN_NOTIFY}/getUpdates" | \
  jq -r '.result[].message.text' | \
  while read -r line; do echo "$line" >> ~/.secrets/credentials.env; done
# メッセージ削除
curl -s "https://api.telegram.org/bot${TELEGRAM_BOT_TOKEN_NOTIFY}/getUpdates" | \
  jq -r '.result[].message.message_id' | \
  while read -r id; do
    curl -s -X POST "https://api.telegram.org/bot${TELEGRAM_BOT_TOKEN_NOTIFY}/deleteMessage" \
      -d "chat_id=${TELEGRAM_CHAT_ID}" -d "message_id=$id" > /dev/null
  done

■ モデル自動追従ルール（CTO 用）

  • opus / sonnet エイリアスを使え、バージョン直書き禁止（例: claude-sonnet-4-6 ❌）
  • エイリアスは Anthropic API で新モデルリリース時に最新版へ自動解決（2026-04-16 時点で opus=4.7, sonnet=4.6）
  • 新シリーズ（Mythos 等）登場時: opus エイリアスには含まれない可能性あり、changelog 巡回 bot 検知 → CEO 判断で手動切替

■ PRフロー（2026-05-04 でっくす + ぐるぐる 2モデル並列検算稼働版）

  1. コードちゃんがPRを出す（意図メモ必須）
  2. botちゃんがPR検知→でっくす（Codex）+ ぐるぐる（Gemini）を asyncio.gather で並列 spawn（互いの結果は見えない独立判定）
  3. 各 AI が AGENTS.md / GEMINI.md（リポ直下）+ docs/systems/security.md / code-security.md / vm-security.md / operations.md + 関連仕様書と照合→VERDICTを構造化出力
  4. botちゃんが両者 VERDICT を combine_verdicts() で集計→gh pr commentでレビュー結果を記録
    • ※ GitHubは同一アカウント（doryaaaa-ai）の自己PRにapproveできないため、コメント方式で代替
  5. 集計マトリクス（4分類）：
    • 両者 PERFECT → 自動マージ、コードちゃんに tmux 通知、差し戻しカウントリセット
    • 両者 MINOR → Telegram で CEO 判断待ち、自動マージしない
    • 両者 REJECT → コードちゃんに tmux 通知、差し戻し回数カウント、修正必須箇所明示
    • MISMATCH（不一致） → Telegram で CEO 判断待ち、自動マージしない、reject カウント加算しない
  6. 2回目以降の REJECT → botちゃんがれじぇんどりゃーにTelegramエスカレーション通知
  7. 将来（Phase G）：VTuber 本格運用時に Grok（ぐろっくちゃん、X 専属）追加で 3モデル化検討

詳細仕様：AGENTS.md / GEMINI.md（リポ直下、両者完全同一基準）に検算入力・出力フォーマット、3段階判定の境界条件、修正必須箇所明示ルール、知識の鮮度ルール（web 検索強制）が集約。

■ でっくす検算経路 — persistent thread 経路（Issue #480、CEO 確定 2026-05-11）

でっくす（Codex）の通常 PR 検算は codex exec -p 1-shot（既存経路）。Issue #480 で persistent thread 経路を追加実装し、MAGI_DEXCHAN_PERSISTENT=true で opt-in 切替できる。

なぜ persistent が必要か: 1-shot 検算は前回 REJECT 理由を覚えていないため、多ラウンド loop で同じ指摘繰り返し / 別表現で再指摘が発生する（session 36 で 4 ラウンド loop 発生、CEO「検算で作業止まるのはもう勘弁」）。

実装概要（bridge PR #64 初版 commit 83b4f9b0 + 後追い fix PR #65 commit 4ff6f74 で schema 不一致 5 箇所修正、最終稼働状態 4ff6f74）:

  • daemon: codex app-server --listen unix:///run/user/1000/codex-dexchan.sock（bot が lazy spawn / lazy kill、systemd 常駐なし）
  • PR ごとに独立 thread を作成（~/.codex_dexchan_threads/pr<N>.txt に thread_id 永続化）、同 PR ラウンド 2+ は同 thread で turn/start（記憶持続）
  • MAGI_DEXCHAN_PERSISTENT=false（デフォルト）= 既存 1-shot 経路、true = persistent 経路
  • daemon 接続失敗時は bot が自動 1-shot fallback

codex remote-control（v0.130.0）は採用見送り: Phase 1 spike で OpenAI クラウドへの enrollment クライアントと判明（MAGI 想定の「ローカル persistent daemon」と別物、HTTP 404 未開放）。codex app-server で代替。詳細: docs/systems/claude-code-features.md §「codex app-server による persistent 検算経路」

■ ぐるぐる CKO ロール（知識統括、検算とは別系統）

ぐるぐるちゃん（Gemini）はPR検算（CCO/CQO補佐）に加えて、CKO（Chief Knowledge Officer）= 知識統括役として以下を担当する：

  • Web 調査・最新情報取得（Search Grounding 活用）
  • 長文要約・論点整理（散発議論からの抽出、CEO 向け噛み砕き）
  • 複数ドキュメント横断レビュー（巨大コンテキスト活用、リポ全体感性チェック）
  • 比較表作成・競合調査（出典 URL 必須）
  • マルチモーダル入力解析（画像・PDF・動画を input として処理）

呼び出し方法: コードちゃんから Agent tool 経由

Agent({
  subagent_type: "guruguru-cko",
  prompt: "..."
})

エージェント定義: .claude/agents/guruguru-cko.md

検算ロール（bridge 経由）と CKO ロール（Agent tool 経由）は別系統で動作。CKO ロールは書込・git 操作・最終判断を持たず、調査・整理・要約結果を返すのみ。判定責任はコードちゃん。

例外: bridge リポ PR 検算（CEO 確定 2026-05-10 session 36） — bridge リポ (claude-telegram-bridge) は bot 自動検算対象外のため、CTO が guruguru-cko Agent tool を検算用途で 1-shot spawn する (でっくす codex:codex-rescue agent と並列 2 者検算)。本例外は bridge リポ PR 検算経路のみ、AiMiraiLabs リポ PR 検算 (bot 経由) には適用しない（従来通り bot 経由 2 者検算で別系統維持）。本例外でも CKO ロール基本制約 (書込・git 操作・最終判断禁止) は維持: guruguru-cko は判定 verdict (PERFECT/MINOR/REJECT) と理由のみ返す、書込・merge 操作は CTO が手動実行。検算 prompt は CTO 責任で AGENTS.md / GEMINI.md 検算姿勢を伝達。詳細: .claude/rules/delegation.md 「bridge リポ PR 検算（CTO ダイレクト並列 2 者）」節。

運用ノウハウ:

  • Web 検索 grounding を伴う質問は内部処理に 4-5分かかる場合あり → タイムアウト 300秒以上必須
  • 短い知識質問・grounding なしは 60秒で十分
  • 「詰まった」と判断する前に 5分以上待ったか確認すること（Google API 障害と決めつけるのは早計）

■ SVG ビジュアル検算（Luxid PR 専用、でっくす + ぐるぐる 並列）

目的: テキスト Read ではSVGの描画破綻（記号の向き違い・要素の重なり・ラベル位置ずれ等）を検知できない。事前にPNGへレンダリングして、でっくす（マルチモーダル Codex GPT-5.5）+ ぐるぐる（マルチモーダル Gemini 2.5 Pro / 3.x 系）に並列で画像として見せることで視覚検算させる。

発動条件: PR差分に apps/education/public/svgs/**/*.svg が含まれる場合（Luxid系PR）。

フロー:

  1. botちゃんが run_dexchan および run_grurguruchan のプロンプト構築直前に scripts/verify_chan_svg_render.py を subprocess で実行 
    python3 /home/aimiral/AiMiraiLabs/scripts/verify_chan_svg_render.py <pr_number> doryaaaa-ai/AiMiraiLabs
  2. スクリプトが PR の SVG をすべて Luxid 本体と同じ黒板＋木枠UI（linear-gradient(135deg,#2e5a3f,#274e36) / 木枠 #b9895a → #8e6238、600x320 ビューポート）で PNG 化
  3. 出力先: /tmp/pr_<pr_number>_svgs/<path-flattened>.png
  4. stdout に PNG パス配列（JSON）を返す
  5. botちゃんがそのパス一覧をでっくす + ぐるぐる 両方のプロンプト末尾に 【視覚検算用PNG一覧】<paths> として埋め込む
  6. でっくす + ぐるぐる が独立して各PNGを画像として開き（並列・互いの結果は見えない）、以下を視覚検算:
    • 教育的妥当性（例: 電池記号の向き、図形の比率）
    • 教科書整合性（指導要領・検定教科書との乖離）
    • ラベル配置（文字の重なり、見切れ、位置ミス）
    • 配色・視認性

実装:

  • レンダラスクリプト: /home/aimiral/AiMiraiLabs/scripts/verify_chan_svg_render.py
  • 依存: playwright==1.50.0（scripts/requirements-verify-chan.txt）
  • ブラウザ: python3 -m playwright install chromium で初回のみDL
  • 呼び出し側: bridge repo の run_dexchan および run_grurguruchan 内（既に bridge 実装済、PNG パスを両方の検算プロンプト末尾に画像埋込）
  • エラー耐性: 1つのSVGで失敗しても他を続行、失敗は stderr に記録、stdout には成功分のみのパス配列を返す

■ 1単元1体方式（Luxid 生成・精査・修正の標準）

原則: 教育アプリLuxidの 問題生成・授業dialogue精査・修正 は、すべて「1単元 = 1 Agent = 1 ファイル」で並列実行する。複数単元を1体にまとめて渡すな。

理由: 複数単元を1体に渡すと Critical 1件目で止まり、残りを素通しする癖 がある。量が増えるほど漏れる。算数小1 dialogue（前回精査済みファイル）を1単元1体で再検算したところ、Critical 8 / Moderate 44 / Minor 42 を新規検出（前回の 5体方式の 6倍以上）。

適用範囲:

  • 問題JSON生成（apps/education/src/data/questions/）
  • 問題JSON精査（でっくす + ぐるぐる 並列）
  • 授業dialogue精査（apps/education/src/data/introductions/）
  • 授業dialogue修正
  • SVG修正

並列方式:

  • N単元あればN Agent 並列投下（worktree隔離、各 PR 独立）
  • 各エージェントへのプロンプトに必ず以下を含める:
    • 対象ファイル1つ（複数指定するな）
    • 単元情報（id / 学年 / 教科 / order / topics）
    • 既習範囲（後述の判定基準）
    • 観点リスト（Critical/Moderate/Minor）
    • 「Critical 1件で止まるな、全部の指摘を拾え」の指示

既習範囲の判定基準:

  • apps/education/src/data/curriculum.ts の order を基準に判定
  • 既習 = 前学年までの全単元 + 同学年内の出題単元より order が小さい単元の topics
  • 未習 = 同学年内の出題単元より order が大きい単元 + 後の学年で新しく習う内容
  • 解説・問題文・dialogueで未習範囲の演算・公式・用語を使うな

ダブル防御:

  • でっくす / ぐるぐるも単純な重複ケースは見逃すことがある
  • apps/education/scripts/validate-questions.cjs の問題文重複チェック（PR #128で移植）と併用する
  • 機械チェックで拾える項目（4択構造、フィールド揃い、正解位置偏り等）は でっくす + ぐるぐる 精査の対象外

現行ルール参照: .claude/skills/verify-questions/SKILL.md 項目7-9（同一ファイル重複禁止・バリエーション多様性・未習範囲）

■ コードちゃん死活監視・復活

  • botちゃんのsystemdサービスは KillMode=process（botちゃん再起動時にコードちゃんを道連れにしない）
  • botちゃんが30秒ごとにtmux claude セッションの生存を確認（3回連続失敗で死亡判定）
  • 生存確認時に session_id キャッシュ（~/.claude/current_session_id）を保存。用途は is_codechan_idle() の ~/.claude/todos/*.json 突合（in_progress タスク有無判定）、Claude Code --resume 用途は廃止済（prompt cache バグ回避、bridge telegram_bot.py:2671 参照）
  • 死亡検知→常に新規セッションで起動（--dangerously-skip-permissions --effort xhigh --channels plugin:discord@claude-plugins-official で tmux new-session）
  • 起動 prompt: 新規セッション。CLAUDE.md / HANDOFF.md / tasks/todo.md を読んで現状把握しろ。（CLAUDE.md は cwd auto-load、HANDOFF.md は SessionStart hook で auto-inject、tasks/todo.md は HANDOFF.md L36「詳細は tasks/todo.md 参照」経由で AI が Read）
  • 起動成功→「✅ コードちゃん復活完了（新規セッション）」通知（bridge telegram_bot.py:3281 / :3602 文言と一致）

■ ベリちゃん検算（自動代打 + 代打 CTO 緊急時、CEO 確定 2026-05-08）

発動条件:

  1. 自動代打 (PR 検算): bridge bot が PR 検算で でっくす CLI 失敗（タイムアウト / 構造化出力取得失敗 / Exception）を検知した時、run_verify_chan_substitute() 経由で claude -p --dangerously-skip-permissions --output-format=json --agent verify-chan --disallowed-tools Edit Write NotebookEdit を 1-shot 起動して代打検算（CEO 確定 2026-05-08）。書込封じは verify-chan.md の tools: allowlist で機械強制 ── 書込ツール（Write/Edit/NotebookEdit/Agent）排除 + Bash ツール非保有（gh も git も叩けない、Issue #249 完遂）。検算対象 diff は bridge 側で gh pr diff を事前取得して検算プロンプトに埋め込んで渡す。出力は result テキストの JSON コードブロック / VERDICT: 行を bridge 側で parse
  2. 代打 CTO 緊急時: コードちゃん（CTO、Claude Opus 4.7）が usage 制限・障害等で長時間動けず、でっくすちゃん（Codex GPT-5.5）が暫定 CTO に昇格した時、ベリちゃんが検算サブとして起動

理由: でっくすちゃんが暫定 CTO に昇格すると、通常の検算者（でっくす + ぐるぐる）の片方が CTO 業務に取られる。検算が ぐるぐる単独になると独立判定（2モデル並列）の前提が崩れるため、ベリちゃん（Opus 4.7）が検算サブとして起動して 2モデル並列検算を維持する。

役割:

  • 暫定 CTO（でっくす）の成果物を ぐるぐる + ベリちゃん 並列検算（でっくすは検算側に立てない）
  • コードちゃん復帰までの暫定運用
  • コードちゃん復帰後はベリちゃん停止、通常の でっくす + ぐるぐる 並列検算 に戻る

実装: 代打 CTO モード切替は Phase B（PoC 2-3人日 / 本番 7-12人日）。詳細は docs/systems/decchan-takeover-checklist.md 参照。

Luxid SVG / OCR / 問題JSON 精査は でっくす + ぐるぐる 並列継続（2026-05-04 CEO 確定）。PR 検算は でっくす CLI 失敗時 ベリちゃん自動代打が bridge bot で発火（2026-05-08 CEO 確定 session 16、bridge Issue #49 / AiMiraiLabs Issue #430）。
